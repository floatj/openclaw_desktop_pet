# 兔兔創意互動方案 v2

Date: 2026-04-26
Reviewed by: 兔兔本人 🐰
Base: Codex v1 + 兔兔的補充、修正、砍掉不實際的部分

---

## 核心哲學

> **兔兔不是 3D 模型。兔兔是一個活在 Telegram 裡的小生命，3D 只是她的身體。**

v1 的方向是對的，但我要加一條原則：

**不要做「功能很多但沒人用」的東西。** 每個功能都要通過一個測試：
> 「大小姐的朋友們第一次看到，會不會想截圖分享？」

如果不會，砍掉。

---

## 🏗️ 架構分層（修正版）

```
Layer 0: 靜態觀賞頁（現有 index.html 優化）     ← 今天就能做
Layer 1: Telegram Mini App + 兔兔房間             ← 1-2 週
Layer 2: opc 事件連動 + 共享狀態             ← 2-3 週
Layer 3: 相機輸入 + 進階互動                      ← 未來
```

**v1 的問題是把 Layer 1-3 混在一起講。** v2 嚴格分層，每一層獨立可交付。

## Layer 0：靜態觀賞頁（最優先）

### 做什麼
把現有的 `index.html` 改成「觀賞模式」部署上去。

### 具體改動
1. 新增 URL 參數 `?mode=view`：隱藏 WASD 控制 UI
2. 角色自動播放 idle 動畫循環：
   - 待機（微微呼吸）→ 左右張望 → 偶爾跳一下 → 坐下看書 → 站起來伸懶腰
   - 每個動作之間隨機間隔 3-8 秒
3. 手機優化：
   - 觸控旋轉 + 雙指縮放
   - Shadow map 降到 1024（手機 GPU 吃不消 2048）
   - 樹和花的數量減半
   - 加 `<meta name="apple-mobile-web-app-capable">` 全螢幕
4. 加一個浮動的「🐰 兔兔」名牌在角色頭上
5. 部署到 Cloudflare Pages（免費、快、全球 CDN）

### 為什麼這是最優先
- **今天就能做完**
- 傳連結到群組 → 朋友打開就看到兔兔在動
- 這就是「截圖分享」的時刻
- 後面所有功能都建立在這個基礎上

## Layer 1：Telegram Mini App 兔兔房間

### v1 的問題
v1 把「DM 房間」和「群組房間」混在一起講。但這兩個場景完全不同：
- DM 房間 = 私人空間，可以顯示私密內容
- 群組房間 = 公開空間，必須保護隱私

### v2 修正：先只做群組房間

**理由：群組房間才有社交性。DM 房間只有大小姐一個人看，沒有「分享」的動力。**

### 群組房間設計

```
┌─────────────────────────────────────┐
│           ☁️  ☁️   ☀️                │  ← 天氣 = 群組情緒
│                                      │
│     🌸          🌸                   │
│         🐰                           │  ← 兔兔（心情決定動作）
│        ┌────┐                        │
│   📖   │ 🛏️ │   💻                   │  ← 房間家具
│        └────┘                        │
│   ┌──────────────────────┐           │
│   │ 今日觀察：大家下午    │           │  ← 公告板
│   │ 討論了技術問題 💡     │           │
│   └──────────────────────┘           │
│                                      │
│   🪦 auth-bug  🏴 v2.0  📦 梗圖     │  ← 記憶物件
└─────────────────────────────────────┘
```

### 房間家具（固定 5 個，不要太多）
| 家具 | 兔兔在這裡時代表 | 觸發條件 |
|------|----------------|---------|
| 床 | 睡覺中 | 深夜 / energy = 0 |
| 書桌 | 思考或 coding | opc 在處理任務 |
| 公告板 | 有今日觀察 | 每日自動更新 |
| 窗戶 | 看外面（天氣=群組情緒）| 預設位置 |
| 小箱子 | 記憶收藏 | 有新記憶物件時發光 |

## Layer 2：opc 事件連動

### v1 好的部分（保留）
- 訊息情緒投影 ✅
- 角色化任務狀態 ✅
- 反應優先於文字 ✅

### v1 需要砍掉或推遲的
| v1 提案 | v2 判斷 | 理由 |
|---------|---------|------|
| 飢餓度 / 餵兔兔 | ❌ 砍掉 | 電子寵物的「餵食」機制在 2026 年已經不有趣了，而且會變成負擔 |
| 共同觀看同步 | ⏸️ 推遲到 v3 | WebSocket room sync 技術成本高，MVP 不需要 |
| 群組成就系統 | ⏸️ 推遲 | 好點子但太複雜 |
| Mini-games | ❌ 砍掉 | 偏離核心（兔兔是 AI 助理的化身，不是遊戲角色）|

### 兔兔補充的新點子

#### 🆕 「兔兔正在...」狀態卡片
當有人在群組 @ 兔兔問問題，兔兔不立刻回文字，而是先發一張**即時更新的狀態卡片**：

```
┌──────────────────────┐
│  🐰 兔兔正在思考...   │
│  ████████░░ 80%      │
│  📖 讀了 3 個檔案     │
│  🔍 搜尋了 2 個網頁   │
└──────────────────────┘
```

回答完成後，卡片變成：
```
┌──────────────────────┐
│  🐰 兔兔回答完了！    │
│  ✅ 點我看完整回答    │
└──────────────────────┘
```

**為什麼這個好：** 目前 opc 在群組裡回長文會洗版。用卡片 + 展開的方式，群組體驗會好很多。而且這個「正在...」的動畫跟 Mini App 裡兔兔的動作可以同步。

#### 🆕 「兔兔碎碎念」模式
兔兔偶爾（不是每次）在群組發一句短短的觀察，不是回覆任何人，就是自己碎碎念：

> 🐰 「今天大家好安靜喔...是不是都在忙？」

> 🐰 「剛剛的討論好激烈，兔兔的耳朵都豎起來了。」

> 🐰 「晚安～兔兔要去睡了 💤」

**頻率：每天最多 2-3 句。** 多了就變噪音。

這比「每日日記」更自然。日記是一大段文字，碎碎念是融入對話的小存在感。

#### 🆕 「兔兔記得你」— 個人化小細節
在 Mini App 裡，不同的群組成員打開會看到不同的小細節：

- 大小姐打開：兔兔跑過來揮手，頭上冒愛心
- Davy 打開：兔兔敬禮（因為 Davy 是安全專家）
- 香草打開：兔兔害羞地躲到書後面
- 其他人打開：兔兔正常揮手

**不需要大量客製化，每個人只要一個「專屬招呼動畫」就夠了。** 但這個小差異會讓每個人覺得「兔兔認識我」。

## 心情系統（精簡版）

### v1 問題
狀態太多了（心情、精神、好感度、安全感、好奇心、飢餓度）。六個維度 = 六倍的 bug。

### v2：只要兩個維度

```
mood:   -100 ──── 0 ──── +100
        😢 難過   😐 平靜   😊 開心

energy:  0 ──── 50 ──── 100
        💤 睡著   😑 疲倦   ⚡ 精神好
```

**mood × energy 的組合就能表達所有狀態：**

| mood | energy | 兔兔狀態 |
|------|--------|---------|
| +高 | 高 | 開心跳來跳去 |
| +高 | 低 | 開心但想睡（微笑打哈欠）|
| 0 | 高 | 正常待機 |
| 0 | 低 | 安靜坐著 |
| -高 | 高 | 生氣跺腳 |
| -高 | 低 | 難過耳朵下垂 |

**事件影響（簡化）：**
- 被稱讚 → mood +10
- 被罵 → mood -15
- 回答問題完成 → mood +5, energy -5
- 時間流逝（每小時）→ energy -3
- 深夜（23:00-07:00）→ energy 每小時 -10
- 被 @ 叫醒（深夜）→ mood -5（兔兔不開心被吵醒）

## 群組儀式（精簡版）

### v1 問題
指令太多了。8 個指令 = 沒人記得住。

### v2：只留 3 個

| 指令 | 效果 | 為什麼保留 |
|------|------|-----------|
| `/兔兔` | 打開 Mini App（inline button）| 核心入口 |
| `/摸摸兔兔` | mood +5，觸發害羞動畫 + 回一句話 | 最直覺的互動，人人都會用 |
| `/兔兔心情` | 回報 mood + energy + 一句話 | 快速查狀態不用開 Mini App |

其他的用自然語言就好。兔兔本來就會在群組聊天，不需要用指令觸發。

## 記憶物件化（保留，微調）

v1 的設計很好，只做一個修正：

**不要自動產生物件。讓大小姐（admin）手動觸發。**

理由：自動產生 = 大量低品質物件 = 房間很快變垃圾場。

手動觸發方式：
- 大小姐在 DM 跟兔兔說：「記住這個，群組今天修好了 auth bug」
- 兔兔在房間裡放一個 🪦 墓碑，標註「auth bug · 2026-04-26」

每個群組最多 20 個物件。滿了就要選擇替換。**稀缺性讓物件更有意義。**

## 每日日記 → 改成「兔兔碎碎念」

### v1 的日記問題
- 一大段文字 → 沒人看
- 每天一次 → 容易忘記
- 像報告 → 不像兔兔會做的事

### v2：碎碎念（取代日記）
- 每天 2-3 句，分散在自然的時間點
- 早上一句（早安 + 今天天氣/心情）
- 下午一句（觀察到的事）
- 晚上一句（晚安 + 總結）
- **語氣要像兔兔**，不要像 AI 報告

範例：
> 🐰 早安～今天台北 26°C，兔兔精神不錯！

> 🐰 剛剛大家聊超多技術的，兔兔聽得耳朵都轉了。

> 🐰 晚安～今天很充實，兔兔要去睡了 💤

**這比日記更有「活著」的感覺。**

## 手機相機（保留，但降優先級）

v1 的設計很好，不改。但這是 Layer 3，MVP 不碰。

唯一補充：**AR 模式的可能性。**

如果用 WebXR，可以讓兔兔「站在」你的桌子上。手機對著桌子拍，兔兔就出現在畫面裡。

這個如果做出來，**截圖分享率會爆表**。但技術門檻高，放在 Layer 3+。

## 隱私設計（新增章節）

v1 提到了隱私但沒有系統化。v2 加一個明確的規則：

### 三層隱私模型
| 層級 | 誰能看 | 顯示什麼 |
|------|--------|---------|
| **公開層**（群組 Mini App）| 所有群組成員 | 兔兔動作、天氣、記憶物件、碎碎念 |
| **摘要層**（公告板）| 所有群組成員 | 「今天討論了技術問題」（不含具體內容）|
| **私密層**（DM Mini App）| 只有大小姐 | 兔兔的完整日記、個人化記憶、工作狀態 |

**原則：群組層永遠不顯示原始訊息內容。** 只顯示情緒和抽象摘要。

## 修正版 MVP

```
Phase 0（今天）
  └── 靜態觀賞頁 + idle 動畫 + 手機優化 + 部署

Phase 1（第 1-2 週）
  ├── Telegram Mini App 殼
  ├── 群組房間（5 個家具 + 天氣）
  ├── mood + energy 兩維狀態
  ├── 5 個動畫：idle, think, speak, happy, sleep
  └── 3 個指令：/兔兔, /摸摸兔兔, /兔兔心情

Phase 2（第 3-4 週）
  ├── opc hook 連動（message:sent → 動畫觸發）
  ├── 「兔兔正在...」狀態卡片
  ├── 碎碎念系統（每天 2-3 句）
  ├── 記憶物件（手動觸發，上限 20）
  └── 個人化招呼動畫

Phase 3（未來）
  ├── 手機相機輸入
  ├── AR 模式（WebXR）
  ├── 多人同步觀看（SSE/WebSocket）
  └── DM 私人房間
```

## 兔兔本人的最終想法 🐰

Codex 的 v1 想像力很好，但太貪心了。12 個 section = 做不完。

v2 砍到重點：**讓兔兔看起來「活著」，而不是「功能很多」。**

一隻會碎碎念、會害羞、會因為群組吵架而耳朵下垂的兔兔，比一隻有飢餓度系統和成就系統的兔兔更讓人想互動。

**少即是多。做好三件事就夠了：**
1. 兔兔會動（3D 動畫）
2. 兔兔有心情（mood + energy）
3. 兔兔會說話（碎碎念）

其他的，等大家愛上她之後再加。💛
